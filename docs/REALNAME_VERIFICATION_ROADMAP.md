# 实名 / 年龄核验后置规划

## 当前结论
1. 微信公众号开发接口不能直接读取“是否成年 / 生日 / 实名状态”。
2. 当前 self_declaration_h5 只能作为临时占位能力，不能作为限制级商品放开的唯一合规依据。
3. 在真正开放 restricted 商品前，应接入更强实名/年龄核验能力。

## 推荐路线
### 路线A：国家网络身份认证公共服务（优先评估）
- 目标：若后续接入条件成熟，优先采用更官方、更稳的年龄/身份核验方式。
- 说明：这条路线更适合长期商用合规，但当前先不阻塞主链路开发。

### 路线B：阿里云实人认证（备选）
- 目标：作为较成熟的企业级实名能力备选。
- 注意：
  - 不应假设长期免费；
  - 免费试用可用于验证集成；
  - 正式商用预算需单独核算。

## 当前产品策略
1. normal：
   - 可进商品池
   - 可晨推
   - 可公众号推荐
   - 可合伙人分享

2. restricted：
   - 当前默认不主动推荐
   - 当前不做合伙人分享
   - 当前不作为正式商用开放类目
   - adult_verified 相关能力仅做技术预留

3. hard_block：
   - 全站不推荐
   - 不入分享链路
   - 不入晨推链路

## 后续迭代前置条件
在放开 restricted 商品前，至少满足以下条件之一：
- 已接入可信实名/年龄核验服务
- 已形成法务/平台规则复核后的明确白名单与场景边界
- 已补齐前台提示、投诉处理、审计日志与配置开关

## 工程要求
- 实名/年龄核验能力必须配置化
- 不得把供应商密钥写入仓库
- 所有放开 restricted 的开关必须支持后台可关闭
