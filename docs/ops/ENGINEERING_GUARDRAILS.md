# AIdeal CPS 工程红线与设计记忆

本文档用于防止后续开发反复覆盖已经验证过的设计。

## 1. 公网路由与链接规则

用户可见链接统一不带 `/api`：

- `/users/test-init`
- `/h5/recommend/{product_id}`
- `/h5/recommend/more-like-this`
- `/promotion/redirect`

生产 Nginx 路由层已经拆成独立模块：

- 生产文件：`/etc/nginx/snippets/aideal_public_backend_routes.conf`
- 仓库模板：`ops/nginx/aideal_public_backend_routes.conf`

该模块必须 include 在静态 `/h5/` 和 `/` 路由之前。
后续如果 H5、京东跳转、用户初始化出问题，先查 Nginx location 优先级，不要先改业务代码。

## 2. 微信菜单交互规则

### 找商品

`找商品` 菜单不是 News 卡片。

设计目标：

- 用文本承接用户；
- 提示用户可以直接输入商品需求；
- 同时推荐 1 个热门、高品质、性价比好的商品；
- 用 emoji 分段展示商品名、价格优势、品类、店铺、推荐理由、放心下单、图文详情、京东快捷下单、更多类似商品。

配置来源：

- `config/wechat_find_product_entry.json`

不要把这类文案常量写死进 Python 代码。

### 今日推荐

`今日推荐` 菜单保持 3 条 News 图文卡片。

设计目标：

- 每次给 3 个商品；
- 与同一用户近期看过的商品去重；
- 点击进入 H5 详情页；
- H5 中再引导京东快捷下单。

## 3. H5 商品详情页规则

H5 详情页不是简单 News 展开页。

H5 有足够空间，应该承接更完整的购买决策信息：

- 商品主图或海报；
- 价格、券后价、可省金额；
- 销量、评价、店铺、类目；
- 推荐理由；
- 与用户消费心理相关的表达：从众心理、损失厌恶、占便宜心理、省心心理、品质/面子心理；
- 明确按钮：`点击我去京东快捷下单`、`更多类似商品`。

## 4. 京东跳转规则

`/promotion/redirect` 必须最终 302 跳转到京东联盟链接或京东短链。

验证标准：

- `GET /promotion/redirect?...` 返回 `302`
- `Location` 指向 `https://u.jd.com/...` 或京东可用链接

如果公网返回官网首页 HTML，优先检查 Nginx 是否把 `/promotion/` 吞到静态站点，不要先改 JD 业务逻辑。

## 5. 用户数据与画像规则

系统未来面向十万级以上用户，所有用户数据必须按单个用户隔离。

必须遵守：

- 推荐去重按用户 openid 的哈希隔离；
- 曝光记录按用户 openid 的哈希隔离；
- 用户画像按用户 openid 的哈希隔离；
- 不允许所有用户共用一个全局去重池；
- 不允许误用公众号 `from_user` / `to_user` 方向；
- 微信真实用户 openid 才是用户身份主键来源。

用户敏感数据必须加密存储：

- 明文 openid 不再写入业务表；
- 用 HMAC hash 做查询索引；
- 用 ciphertext 保存可恢复身份；
- 昵称、偏好、最近查询等画像字段也应逐步加密。

## 6. 配置与常量规则

业务文案、按钮文案、阈值、路径模板、标签、推荐理由模板，不要硬编码进 Python。

优先级：

1. 数据库 / 管理后台；
2. `config/*.json`；
3. `app/core/*_config.py`；
4. 最后才是代码内极少数默认兜底。

## 7. 修改节奏规则

线上生产环境修复时，必须小步推进：

1. 一个问题一个问题修；
2. 每次只改少量文件；
3. 先本地 compile；
4. 再做单点 smoke；
5. 再做 route-level smoke；
6. 再重启服务；
7. 再 post-restart smoke；
8. 最后 commit + push。

不要把欢迎语、H5、Nginx、用户加密、推荐去重、JD 跳转放在一个大补丁里一起改。

## 8. 当前不可回退结论

- no-api 公网路径是最终口径；
- 路由层独立模块化是最终口径；
- 找商品菜单文本承接是最终口径；
- 今日推荐 News 卡片是最终口径；
- 用户画像和去重按单用户隔离是最终口径；
- 用户身份加密存储是最终口径；
- 文案和配置不得硬编码是最终口径。

## 9. 历史文档读取规则

`memory/HANDOFF.md`、`docs/PROJECT_HANDOFF.md`、`docs/WECHAT_RECOMMEND_SYSTEM_DESIGN.md`、`docs/system_design.md` 中存在按时间追加的历史记录。

读取这些文档时必须遵守：

1. 以最新日期块和本文档为准；
2. 更早日期块如果与本文档冲突，视为历史方案，不得作为当前实现依据；
3. 不得因为早期记录中出现“找商品图文化”“/api 路径”“全局去重”等表述，就回退当前方案；
4. 当前方案必须以生产已验证链路为准。

## 2026-04-26 大模型文案打磨语气红线

本项目所有大模型打磨的推荐理由、商品说明、H5 转化文案、公众号回复文案，必须遵守官方客服级表达口径。

新增配置文件：

- `config/llm_copy_guardrails.json`

强约束：

1. 严禁使用毒舌、低俗、轻佻、怪诞、阴阳怪气、擦边、羞辱、攻击、过度娱乐化等语气。
2. 智省优选不是娱乐号，也不是猎奇内容号；定位是官方客服级 AI 京东导购助手。
3. 文案目标是帮助用户获得更值、更省、更省心的消费决策，并合规促进佣金商品成交。
4. 可以借鉴从众心理、损失厌恶、占便宜心理、面子心理，但表达必须克制、体面、专业，不制造压迫感。
5. 推荐理由必须围绕真实可解释因素：券后价、日常价、预计可省、销量、评价、店铺确定性、品牌/用途匹配、省时间、少踩坑。
6. 不承诺绝对最低价，不夸大质量，不承诺固定收益，不诱导非理性消费。
7. 后续接入 LLM 文案生成时，系统提示词和后处理审核都必须读取或等价实现该 guardrail。

## 2026-04-26 JD价格新鲜度缓存策略

本次固化商品价格实时读取策略：

1. H5 商品详情页首次访问时，如果商品价格校验已过期，才触发 JD 实时价读取。
2. 默认同一商品 72 小时内不重复读取 JD 实时价，避免浪费京东接口调用次数。
3. 如果 JD 实时价接口没有返回严格有效价格，也记录本次检查时间，避免每次打开都重复调用。
4. 只有拿到严格校验价时，H5 才展示“优惠价 / 京东官网价 / 立省”等强价格对比口径。

6. 后台刷新脚本：`scripts/refresh_stale_product_prices.py`。
7. 配置文件：`config/jd_price_refresh_policy.json`，默认 72 小时刷新窗口，可后续改为 24 小时或 72 小时。

## 2026-04-26 价格刷新并入高质量推荐后台任务

本次修正价格新鲜度策略：

1. 不单独维护一套孤立后台任务，真实价格刷新并入现有高质量推荐商品后台链路。

5. `Product` ORM 映射 `price_verified_at / purchase_price / basis_price / basis_price_type / is_exact_discount`，确保价格校验时间能写回。
6. 未拿到严格校验价时，不展示可能和京东落地页冲突的强价格对比，只引导用户点击京东页查看实时券价。

## 2026-04-27 推荐价格刷新闭环最终口径

本次已将京东价格刷新闭环并入推荐主链路，避免用户侧 H5 价格与京东下单页价格不一致。

最终口径：

1. 新入库佣金商品在 `jd_product_sync_service.normalize_jd_item()` 阶段，直接从京东 `priceInfo` 写入价格快照：
   - `purchase_price`
   - `basis_price`
   - `basis_price_type`
   - `is_exact_discount`
   - `price_verified_at`

2. 后台推荐主任务 `run_nightly_catalog_refresh.py` 不只刷新商品池和白名单，也会调用价格补齐逻辑，持续刷新推荐池商品价格。

3. 推荐排序优先使用已刷新价格的商品；已通过精确价差验证的商品优先级更高。

4. H5 价格展示强约束：
